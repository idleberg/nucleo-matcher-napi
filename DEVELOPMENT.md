# Developer Guide

## Setup

```sh
pnpm install
pnpm build      # napi build --platform --release
pnpm test       # vitest
pnpm bench      # clones a pinned kernel tree on first run, then benchmarks
```

`rust-toolchain.toml` pins **1.90.0**, matching the toolchain the published
`nucleo-matcher-wasm` artifact was built with, so the wasm/native comparison in the
benchmarks is apples to apples. Older toolchains will fail on `unicode-segmentation`, which
requires 1.85+.

Commit `Cargo.lock`. This crate produces binaries across seven triples; the CI matrix must
resolve identical dependencies on all of them.

## npm lifecycle scripts

| script           | when it runs           | what it does                                                                                                                                                                     |
| ---------------- | ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `build`          | manual                 | `napi build --platform --release` — compiles the Rust crate into a `.node` binary for the current platform                                                                       |
| `version`        | `npm version <semver>` | `napi version` — syncs the new version into every platform package under `packages/` (e.g. `nucleo-matcher-napi-darwin-arm64`) so their `package.json` versions stay in lockstep |
| `prepublishOnly` | `npm publish`          | `napi prepublish -t npm` — rewrites `index.js` to load the correct platform binary from `optionalDependencies`                                                                   |
| `artifacts`      | CI                     | `napi artifacts` — collects per-platform `.node` files from CI matrix jobs into the `packages/` directory tree                                                                   |

## Benchmarks

94,841 Linux kernel file paths (avg 39 chars), `maxResults: 200`, p50 of 25 reps after 10
warmups, **one child process per measurement**.

Reproduce with `pnpm bench`. Caveats:

- Never benchmark two engines in one process. Doing so produced a 3x phantom slowdown on a
  zero-hit query, purely from GC and memory pressure between the WASM heap and the native
  allocator. The benchmark spawns a child per measurement for this reason.
- `score()` is the one entry point where a WASM build beats N-API (~150ns vs ~290ns per
  call), because each string argument costs a two-pass `napi_get_value_string_utf8`. Batch
  through `matchIndexed` in hot paths.
- On Apple Silicon, zadeh only ships x64 prebuilds. The benchmark detects the hardware
  architecture via `sysctl` and selects the right Node binary per library (arm64 for
  nucleo-matcher-napi, x64 via fnm for zadeh). This works regardless of whether the
  orchestrator itself is launched from an arm64 or x64 shell.

Benchmark dependencies (zadeh, nucleo-matcher-wasm) live in the `benchmark/` workspace so
they never leak into the published package.

## Design notes

**Interning.** The haystack is converted to `Utf32String` once, in the constructor or
`setItems`. Per query only the pattern crosses the boundary, and results come back as
`Uint32Array`. Boundary cost is under 1.5% of a query on a 95k corpus.

**Ordering is deterministic.** Results sort by descending score, ties broken by ascending
haystack index. Sorting on score alone leaves `sort_unstable` free to reorder tied results
between calls — on a large corpus most results tie, and the jitter is visible in a UI.

**`maxResults` skips the sort, not just the copy.** It uses `select_nth_unstable_by`, which
is O(n) average, then sorts only the retained _k_. On 79k hits that is ~1.8ms → ~0.2ms.

**Scoring dominates.** On this corpus, 89–98% of a query is the scoring loop. Sorting is
2–10%. Marshalling is noise. If you are optimizing, optimize the matcher or run it on more
threads; there is nothing to win around the edges.

## Project structure

```
├── src/            Rust crate (N-API bindings via napi-rs)
├── packages/       per-platform packages published as optionalDependencies
├── benchmark/      pnpm workspace with its own devDependencies (zadeh, wasm)
├── tests/          vitest tests
├── index.js        generated by napi — loads the right .node binary
├── index.d.ts      generated by napi — TypeScript declarations
├── mise.toml       task runner and tool versions (node, lefthook)
└── lefthook.toml   pre-commit hooks (cargo fmt + clippy on *.rs)
```
